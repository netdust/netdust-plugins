---
name: constitution-bridge
description: Generate or refresh a project's spec-kit constitution (.specify/memory/constitution.md) as a VIEW over the netdust governance sources — RULES.md (non-negotiables), SOUL.md (voice/altitude), and the project's ARCHITECTURE-INVARIANTS.md — instead of authoring an independent constitution. Keeps one source of truth so spec-kit's /speckit.plan and /speckit.implement consult the SAME governance the harness already enforces. Use when setting up spec-kit on a project (replaces /speckit.constitution), or after RULES.md / SOUL.md / ARCHITECTURE-INVARIANTS.md change. NOT a place to invent new rules — if a new non-negotiable is needed it goes in RULES.md first, then this skill regenerates the view.
---

<objective>
spec-kit's `/speckit.constitution` wants to author `.specify/memory/constitution.md` — the document its planner and implementer consult for "code quality, testing standards, governance." The netdust harness already has that governance, split across three authored sources:

- `RULES.md` — universal non-negotiables (secrets, `vendor/`, branch discipline, deploy confirmation…).
- `SOUL.md` — voice and altitude (pushback over flattery, YAGNI, name the trade-off behind "best practice").
- `<project>/ARCHITECTURE-INVARIANTS.md` — the convergence points (where each cross-cutting property is decided).

Authoring a *fourth* governance doc forks the source of truth — the two will drift, and a reviewer can no longer tell which one binds. This skill instead **generates the constitution as a derived VIEW** over those three, with a banner that says so. One source of truth; spec-kit consults the same rules the harness enforces.
</objective>

<extremely_important>
This skill does NOT invent governance. The constitution it writes contains only content **traceable to a line in RULES.md, SOUL.md, or ARCHITECTURE-INVARIANTS.md**. If the planning work reveals a missing rule, the fix is: add it to `RULES.md` (or the invariants doc), THEN re-run this skill. A constitution principle with no upstream source is a fork — delete it and put it where it belongs.

The generated file is **derived, not authored**. It carries a `<!-- GENERATED -->` banner and must never be hand-edited; edits go to the upstream source and the file is regenerated.
</extremely_important>

<process>

**Step 1 — Locate the three sources.**
- `RULES.md`: the netdust-core one (always) PLUS the loaded stack sub-plugin's `RULES.md` (e.g. `netdust-wp/RULES.md`) if present — stack rules extend core.
- `SOUL.md`: `netdust-core/SOUL.md`.
- `ARCHITECTURE-INVARIANTS.md`: at the project root. If it does not exist and the project has any multi-actor / tenancy / data-access surface, note that and recommend `/architecture-invariants audit` — but do not block; emit the constitution with an "(invariants doc not yet authored)" note in that section.

**Step 2 — Map each source to a constitution article.** Do not paraphrase loosely — quote or tightly summarize, and cite the source heading so the trace is checkable.

| Constitution article | Sourced from |
|---|---|
| I. Non-negotiables | `RULES.md` (core + stack), verbatim rule list |
| II. Coding standards | `RULES.md` code section + the stack's standards (PSR-12/WPCS for WP, the project's eslint/prettier config) — this is the article `standards-gate` enforces |
| III. Testing standards | `testing-workflow` discipline summary (risk tiers, Tier-A RED-first, denial paths) — reference, do not restate the whole skill |
| IV. Architecture invariants | the project's `ARCHITECTURE-INVARIANTS.md` convergence points |
| V. Voice & altitude | `SOUL.md` (pushback, YAGNI, name the trade-off) |

**Step 3 — Emit `.specify/memory/constitution.md`** using `<output_template>`. Include the GENERATED banner, the source list with paths, and the generation date.

**Step 4 — Report the trace.** List which source fed which article, and flag any article with a missing source (e.g. invariants doc absent) so the human can fill the gap upstream.

</process>

<output_template>

```markdown
<!-- GENERATED by netdust-agent:constitution-bridge — DO NOT EDIT.
     This file is a VIEW over the netdust governance sources. Edit the source, then
     regenerate. Sources:
       - <path>/RULES.md  (+ stack RULES.md if loaded)
       - netdust-core/SOUL.md
       - <project>/ARCHITECTURE-INVARIANTS.md
     Generated: <DATE> -->

# Project Constitution

## Article I — Non-negotiables (from RULES.md)
[verbatim rule list, core + stack]

## Article II — Coding standards (enforced by netdust-agent:standards-gate)

[risk-tier summary: Tier A = RED-first behavioral test incl. denial path for

## Article IV — Architecture invariants (from ARCHITECTURE-INVARIANTS.md)

[pushback over flattery; YAGNI; no infrastructure before validated need; name the trade-off

```

</output_template>

<red_flags>

| Thought | Reality |
|---|---|
| "I'll just write a sensible constitution from scratch" | That forks governance. The constitution is a VIEW. Every line traces to RULES/SOUL/invariants or it does not belong. |
| "RULES.md is missing a rule the plan needs — I'll add it to the constitution" | Backwards. Add it to `RULES.md`, then regenerate. The constitution never holds rules its sources don't. |
| "I'll hand-edit constitution.md to tweak wording" | It is GENERATED. Edit the source. A hand-edit is erased on the next regenerate and silently diverges until then. |
| "No ARCHITECTURE-INVARIANTS.md, I'll skip Article IV" | Emit the article with the "(not yet authored)" note and recommend the audit — a silent omission hides that the project has no named convergence points. |

</red_flags>

<success_criteria>
1. `.specify/memory/constitution.md` exists, carries the GENERATED banner + source paths + date.
2. Every article traces to a named source heading; no article contains invented governance.
3. Article II names the concrete coding standard (`standards-gate`'s enforcement target).
4. The trace report lists source→article and flags any missing source.
</success_criteria>

<integration>

| Skill / artifact | Relationship |
|---|---|
| spec-kit `/speckit.constitution` | **REPLACES.** Run this instead — it produces the same file as a view, not a fork. |
| `netdust-spec-templates` (plan override) | **CONSUMER.** The plan template's "Constitution check [GATE]" verifies the approach against this file. |
| `netdust-agent:standards-gate` | **CONSUMER.** Enforces Article II (coding standards) at task close. |
| `netdust-agent:testing-workflow` | **REFERENCED.** Article III summarizes its tiers. |
| `netdust-agent:architecture-invariants` | **SOURCE.** Article IV is its convergence points. |
| `RULES.md` / `SOUL.md` | **SOURCES.** Articles I, II, V. Edit here, regenerate the constitution. |

</integration>
